# 代码注入研判

## 目标
判断代码注入的告警是真实攻击，还是业务行为误报。

## 范围
- 系统命令注入
- ping命令注入
- whoami命令注入
- 通用系统命令注入（ls、nc、df、netstat等）
- OS命令执行

## 输入
| 参数 | 说明 |
|------|------|
| `${alert:name}` | 告警名称 |
| `${alert:severity}` | 告警等级 |
| `${alert:direction}` | 告警访问方向 |
| `${alert:attackResult}` | 攻击结果 |
| `${alert:httpRequest}` | http请求内容 |
| `${alert:httpResponse}` | http响应内容 |
| `${alert:url}` | 告警URL路径 |
| `${alert:processPath}` | 告警进程路径 |

## 输出
| 参数 | 说明 |
|------|------|
| `${alert:triageResult}` | 研判结果 |
| `${alert:triageCoT}` | 研判思维链 |

## 工具
- **SOC工具**: `${soc-mcp}`
  - `${soc-getAlert}` - 获取告警信息
  - `${soc-getHttpLog}` - 获取HTTP日志
  - `${soc-getAsset}` - 获取资产信息
  - `${soc-getTcpLog}` - 获取TCP日志
- **EDR工具**: `${edr-mcp}`
  - `${edr-getFileContent}` - 获取文件内容

## 流程

### 正报判断

#### 1. 告警类型分析
- **1-1.** 识别告警类型
  - 从 `${alert:name}` 识别告警类型（ping命令注入、whoami命令注入等）
  - 根据告警描述内容检索对应关键词
  - 如果是通用系统命令注入，尝试搜索常见命令：`ls`、`nc`、`df`、`netstat`、`cat`、`id`、`uname`等

#### 2. 请求特征分析
- **2-1.** 分析请求方式
  - 从 `${alert:httpRequest}` 识别请求方法（GET/POST）
  - 检查请求参数中是否存在命令注入特征

- **2-2.** 分析请求参数
  - 识别是否存在命令连接符: `;`、`|`、`||`、`&&`、`&`、`` ` ``、`$()`
  - 识别是否存在命令注入payload: `$(command)`、`` `command` ``、`; command`、`| command`
  - **注意**：命令可能存在于某个请求头字段中被base64编码，需要单独解码查看

- **2-3.** 分析请求体特征
  - 识别是否存在常见的系统命令: `ping`、`whoami`、`netstat`、`ifconfig`、`ipconfig`、`cat`、`ls`、`dir`、`nc`、`wget`、`curl`
  - 识别是否存在base64编码的命令执行特征

#### 3. 响应特征分析
- **3-1.** 分析响应内容
  - 从 `${alert:httpResponse}` 识别是否包含命令执行结果特征
  - 识别是否包含系统信息（用户名、主机名、系统版本等）
  - 识别是否包含网络信息（网络连接、端口状态等）
  - 识别是否包含文件内容或目录列表

- **3-2.** 分析响应状态
  - 检查 `${alert:attackResult}` 是否为成功
  - 检查HTTP状态码是否为200
  - **关键判断**：响应体中是否返回对应命令的执行结果

#### 4. 关联告警分析
- **4-1.** 分析相关告警
  - 通过 `${soc-getAlert}` 获取该目的IP近24小时的告警
  - 判断是否存在同一源IP的多次命令注入告警
  - 判断是否存在漏洞利用等前置告警

### 误报判断

#### 1. 资产信息分析
- **1-1.** 分析受害资产
  - 通过 `${soc-getAsset}` 获取该目的IP的资产信息
  - 判断 `${alert:direction}` 是否为内对内（内对内需重点分析源IP）

#### 2. 请求特征分析
- **2-1.** 分析请求内容
  - 从 `${alert:httpRequest}` 识别请求内容是否存在明显业务特征
  - 识别是否存在正常的业务参数和值
  - 判断请求URL是否为已知的业务接口路径
  - **关键字误判**：检查命中的关键字是否只是URL文件名的一部分（如`cardinfo`只是文件名，非命令）

- **2-2.** 分析URL路径
  - 从 `${alert:url}` 识别是否为常见的业务文件路径
  - 判断文件名中是否包含被误判为命令的关键字

#### 3. 响应特征分析
- **3-1.** 分析响应内容
  - 从 `${alert:httpResponse}` 识别响应内容是否为正常的业务响应
  - 判断响应是否为JSON、HTML等正常格式
  - **关键判断**：响应体中是否未发现对应命令的执行结果

#### 4. 历史流量分析
- **4-1.** 分析HTTP请求历史
  - 通过 `${soc-getHttpLog}` 获取该源IP近7天的http请求日志
  - 判断该行为日志是否经常存在
  - 判断是否存在其他攻击行为

- **4-2.** 内对内场景分析
  - 针对源IP进行全视角分析
  - 判断是业务行为还是内网失陷发起的横向攻击

#### 5. 历史告警分析
- **5-1.** 分析历史告警
  - 通过 `${soc-getAlert}` 获取该源IP近7天的告警
  - 判断是否存在相同URL的多次误报告警
  - 判断该告警是否为已知业务触发的规律性告警

#### 6. 常见误报场景
- **6-1.** 业务误报场景
  - 客户自身业务误报（业务参数包含命令关键字）
  - 乱码误报（二进制数据被误判）
  - 关键字误判（URL/参数中包含命令字符串）
  - 日志传输行为导致的误报
